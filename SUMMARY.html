The Flaky Network — Architecture Summary

Overview

This solution reliably streams 160-byte audio frames across a hostile UDP relay using immediate packet duplication. The sender transmits each frame twice in rapid succession; the receiver deduplicates and forwards immediately. No feedback channel, no retransmissions, no jitter buffer.

Data Flow

harness source ──47010──▶ SENDER ──47001──▶ relay ──47002──▶ RECEIVER ──47020──▶ harness player │ │ │ │ sends each frame │ deduplicates │ twice (back-to-back) │ forwards immediately │ │ └── no feedback used ───────────────┘

Sender Design

When the harness delivers frame i (4-byte big-endian seq + 160-byte payload), the sender:

  1. Forwards the packet to the relay as-is (164 bytes).
  2. Immediately sends a second identical copy.

The two copies are processed by the relay independently — each gets its own drop decision, its own random delay, and its own duplication chance. A frame is only lost if both copies are dropped or arrive after the deadline.

Budget Management

The bandwidth cap is 2.0× the raw stream. With 164 bytes per packet and 160-byte payloads, the 4-byte header overhead limits us: 

raw         = n_frames × 160 bytes
cap         = 2.0 × raw
max_packets = floor(cap / 164)
max_dups    = max_packets − n_frames

For a 30-second run (1500 frames): max_packets = 2926, max_dups = 1426. The last 74 frames are sent without duplication. Their contribution to the overall miss rate is small: 74 × loss_rate / 1500.

Receiver Design

The receiver's logic is deliberately minimal:

  1. Receive a packet from the relay on port 47002.
  2. Extract the 4-byte sequence number.
  3. If this seq has not been delivered yet, forward the packet to the harness player on port 47020 and mark it delivered.
  4. Otherwise, discard (deduplicate).

There is no jitter buffer or playout timer. The harness player only checks whether each frame arrived before its deadline — delivering early is strictly better. A simple boolean array (delivered[seq]) handles deduplication in O(1).

Why Not ARQ (Retransmission)?

Automatic Repeat reQuest requires a feedback path through the relay. Both the NACK and the retransmitted frame must cross the hostile network, incurring at least 2× the one-way delay. With relay delays up to 80ms, the round trip is 40–160ms — often exceeding the playout deadline. FEC or duplication avoids this penalty entirely by sending redundancy proactively.

Why Not XOR FEC?

XOR forward error correction (parity over pairs of frames) was considered and prototyped. It uses less bandwidth (1.5× vs 2.0×) but has a timing problem: the parity for frames (2k, 2k+1) cannot be sent until frame 2k+1 arrives, 20ms after frame 2k. If frame 2k's original is dropped, recovery depends on the parity arriving before 2k's deadline — which often fails when the relay delay is close to the playout budget. Immediate duplication avoids this because both copies of every frame are sent at the same time.

Choosing delay_ms

The playout delay must exceed the maximum one-way relay delay to avoid systematic timing misses. A 2ms margin absorbs processing overhead across the five-process pipeline (harness source → sender → relay → receiver → harness player). Rule of thumb: 

delay_ms = max_relay_delay_ms + 2

Limitations

  • Burst losses: both copies are sent back-to-back and may be processed by the relay during the same loss burst. A Gilbert-Elliott model with long bursts can defeat duplication.
  • High loss rates: above ~7%, the 74 unprotected tail frames contribute more than 0.3% to the miss rate, shrinking the margin for duplicated frames.
  • Delay spikes: any spike exceeding delay_ms causes a miss regardless of duplication — only increasing delay_ms helps.
